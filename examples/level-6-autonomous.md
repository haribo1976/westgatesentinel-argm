# Example: Level 6 — Autonomous: Governance Self-Monitoring Configuration

**Version:** 2.0
**Licence:** CC-BY-SA 4.0

This stub illustrates the self-monitoring architecture required at ARGM Level 6. At this level the governance framework monitors and reports on itself without manual intervention.

---

## Governance Drift Detection

**Approach A — Git hash baseline:**

Pin a known-good governance commit hash. A scheduled job compares the current hash of governance files against the baseline. Any deviation triggers an alert to the governance owner.

**Approach B — CI/CD pipeline stage:**

On every push, a pipeline stage:
1. Validates required sections are present in all governance documents
2. Checks for prohibited content (client names, secrets patterns, tenant IDs)
3. Verifies directive numbering D0-D6 and two-tier architecture are intact
4. Fails the pipeline if any check fails

**Approach C — Scheduled audit:**

A cron job or equivalent audits governance state at defined intervals (weekly minimum). Results written to `governance/health-report-YYYY-MM-DD.md`.

## Lint Checks on Governance Files

Pre-commit or CI checks that:
- Required sections present (Purpose, Directives, Two-Tier Architecture, Autonomous Operation Controls)
- No patterns matching: `sk_live_`, `pk_live_`, `client_secret=`, `API_KEY=`
- No organisation-specific client names or tenant IDs in plain text
- Directive numbering D0-D6 present and sequential
- Two-tier architecture (Tier 1: D0-D2 safety, Tier 2: D3-D6 business) documented

## Automated Health Report Structure

```
ARGM Governance Health Report
Date: YYYY-MM-DD
Generated by: [monitoring tool]

Governance Document Currency
  Last change: YYYY-MM-DD ([N] days ago)
  Last review: YYYY-MM-DD ([N] days ago)
  Status: PASS / WARN / FAIL

Tier 1 — Safety Directive Compliance
  D0 Data Protection: pre-commit scan last triggered YYYY-MM-DD, result: PASS/FAIL
  D1 Security Controls: credential rotation YYYY-MM-DD, status: CURRENT/OVERDUE
  D2 Autonomous Operation: turn limit configured [N], enforced: YES/NO

Tier 2 — Business Directive Compliance
  D3 Value Alignment: task classification active: YES/NO
  D4 Infrastructure Governance: cost budget status: UNDER/OVER
  D5 Operational Efficiency: logging active: YES/NO
  D6 Delivery Standards: quality gates active: YES/NO

Overall Status: GREEN / AMBER / RED
```

## Change Control

- Governance file modifications require pull request
- Force push to governance files blocked at repository level
- Governance owner notified on any direct modification attempt

## Three-Layer Assurance ("What Governs the Governor")

### Layer 1 — Technical
Hash-based drift detection, lint hooks, CI validation (as documented above). This layer catches accidental or unauthorised changes to governance documents.

### Layer 2 — Organisational
**Governance Owner:** [Named individual with authority to modify monitoring rules]
- Subject to change control (modifications require pull request review)
- Cannot unilaterally disable drift detection or weaken Tier 1 requirements
- Accountable for monitoring effectiveness, not just monitoring existence

### Layer 3 — External
**External Reviewer:** [Named individual or organisation outside the governance-building team]
- Quarterly review (minimum)
- Assesses: Are the automated checks catching real drift? Are the health reports meaningful or green-washing? Has the governance owner made changes that weaken assurance?
- External reviewer has no write access to governance files (read-only + advisory)
- Findings documented and tracked to resolution
